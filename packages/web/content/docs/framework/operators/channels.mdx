## Quick Example

```ts
import { z } from 'zod';
import { channel, step } from '@noetic-tools/core';

const notifications = channel('notifications', {
  schema: z.object({ message: z.string(), level: z.enum(['info', 'warn', 'error']) }),
  mode: 'queue',
  capacity: 100,
});

const producer = step.run({ id: 'produce', execute: async (input: string, ctx) => {
  await ctx.send(notifications, { message: input, level: 'info' });
  return input;
}});

const consumer = step.run({ id: 'consume', execute: async (_input: unknown, ctx) => {
  const note = await ctx.recv(notifications, { timeout: 5_000 });
  console.log(`[${note.level}] ${note.message}`);
  return note;
}});
```

Channels provide typed, schema-validated communication between steps, between loop iterations, and between your application and a running agent. They replace ad-hoc state mutation with explicit message passing.

## Creating Channels

Use the `channel()` builder to define a channel. Every channel has a name, a Zod schema, and a mode:

```ts
import { z } from 'zod';
import { channel } from '@noetic-tools/core';

const progress = channel('progress', {
  schema: z.number().min(0).max(100),
  mode: 'value',
});
```

### Builder Options

| Option | Type | Required | Description |
|---|---|---|---|
| `schema` | `ZodType<T>` | Yes | Zod schema that validates every value sent through the channel. |
| `mode` | `'value' \| 'queue' \| 'topic'` | Yes | Delivery semantics (see below). |
| `capacity` | `number` | No | Maximum buffer size for queue mode. Defaults to 1000. |
| `external` | `boolean` | No | When `true`, creates an `ExternalChannel` that outside code can write to. |

## Channel Modes

### Value Mode

Holds the latest value. Readers always get the most recent write. If no value has been sent yet, `recv` blocks until one arrives.

```ts validate=none
const config = channel('config', {
  schema: z.object({ temperature: z.number() }),
  mode: 'value',
});

// Writer sets the value (value-mode sends always resolve immediately)
await ctx.send(config, { temperature: 0.7 });

// Any reader gets the latest value immediately
const current = ctx.tryRecv(config);
// { temperature: 0.7 }
```

Best for: shared configuration, latest-status indicators, feature flags.

### Queue Mode

FIFO buffer. Each value is consumed by exactly one reader. Values accumulate up to `capacity`; when full, `ctx.send` parks (back-pressure) until a consumer dequeues an item — see [Back-Pressure](#back-pressure) below. If the queue is empty, `recv` blocks until a value arrives.

```ts validate=none
const tasks = channel('tasks', {
  schema: z.object({ url: z.string().url() }),
  mode: 'queue',
  capacity: 50,
});

// Producer adds work (resolves immediately while below capacity)
await ctx.send(tasks, { url: 'https://example.com/page1' });
await ctx.send(tasks, { url: 'https://example.com/page2' });

// Consumer takes one item at a time
const task = await ctx.recv(tasks);
// { url: 'https://example.com/page1' }
```

Best for: work queues, task distribution, ordered pipelines.

### Topic Mode

Pub/sub broadcast. Every `recv` call that is waiting at the time of `send` receives the value. Values are not buffered -- if nobody is listening, the message is lost. `tryRecv` always returns `null` in topic mode.

```ts validate=none
const events = channel('events', {
  schema: z.object({ type: z.string(), payload: z.unknown() }),
  mode: 'topic',
});

// Multiple listeners can await simultaneously
const listener1 = ctx.recv(events, { timeout: 10_000 });
const listener2 = ctx.recv(events, { timeout: 10_000 });

// Both receive the same value (topic sends always resolve immediately)
await ctx.send(events, { type: 'update', payload: { status: 'done' } });
```

Best for: event broadcasting, fan-out notifications, real-time updates.

## Channel Interface

```ts validate=none
interface Channel<T> {
  readonly name: string;
  readonly schema: ZodType<T>;
  readonly mode: 'value' | 'queue' | 'topic';
  readonly capacity?: number;
}
```

## Sending and Receiving

Channels are accessed through the [Context](/docs/framework/context) or [AgentHarness](/docs/framework/runtime):

### From a Step (via Context)

| Method | Signature | Description |
|---|---|---|
| `ctx.send` | `send<T>(channel: Channel<T>, value: T): Promise<void>` | Send a value. Resolves immediately unless a queue channel is at capacity, in which case it parks until a consumer frees a slot (default 30s timeout → `channel_timeout`; abort → `cancelled`). |
| `ctx.recv` | `recv<T>(channel: Channel<T>, opts?: { timeout?: number }): Promise<T>` | Wait for a value. Throws `channel_timeout` on expiry (default 30s). Rejects with `cancelled` if the context is aborted while waiting — even with `timeout: 0`. |
| `ctx.tryRecv` | `tryRecv<T>(channel: Channel<T>): T \| null` | Non-blocking read. Returns `null` if nothing is available. |

### From the AgentHarness

The agent harness provides the same three methods, taking an additional `ctx` parameter:

```ts validate=none
await harness.send(channel, value, ctx);
const val = await harness.recv(channel, ctx, { timeout: 5_000 });
const maybe = harness.tryRecv(channel, ctx);
```

## External Channels

External channels let code outside the agent (your application, an API handler, a UI) push values into a running execution.

```ts
import { z } from 'zod';

const userInput = channel('user-input', {
  schema: z.string(),
  mode: 'queue',
  external: true,
});
```

The `external: true` option changes the return type to `ExternalChannel<T>`, which enables `getChannelHandle`:

### ChannelHandle

```ts validate=none
interface ChannelHandle<T> {
  send(value: T): void;
  readonly closed: boolean;
  readonly channel: Channel<T>;
}
```

Get a handle from the agent harness using the execution ID:

```ts validate=none
const handle = harness.getChannelHandle(userInput, ctx.id);

// From your API handler, UI callback, etc:
handle.send('What is the weather in Tokyo?');

// Check if the execution is still running
if (handle.closed) {
  console.log('Execution has ended');
}
```

| Property / Method | Description |
|---|---|
| `send(value)` | Push a value into the channel. Throws `channel_closed` if the execution has ended. |
| `closed` | Whether the execution associated with this handle has been closed. |
| `channel` | Reference back to the channel definition. |

### External Channel Example

A typical human-in-the-loop pattern:

```ts validate=none
import { z } from 'zod';
import { channel, step, loop, until } from '@noetic-tools/core';

const approval = channel('approval', {
  schema: z.enum(['approve', 'reject']),
  mode: 'queue',
  external: true,
});

const askForApproval = step.run({ id: 'ask-approval', execute: async (plan: string, ctx) => {
  // Send plan to user via your app layer...
  const decision = await ctx.recv(approval, { timeout: 300_000 }); // 5 min
  return decision;
}});

// Meanwhile, from your server:
const handle = harness.getChannelHandle(approval, ctx.id);
handle.send('approve'); // Unblocks the step
```

## Schema Validation

Every value sent through a channel is validated against its Zod schema at runtime. This catches type mismatches, malformed data, and contract violations early:

```ts validate=none
const scores = channel('scores', {
  schema: z.number().int().min(0).max(100),
  mode: 'queue',
});

await ctx.send(scores, 85);    // OK
await ctx.send(scores, 101);   // Throws ZodError: Number must be <= 100
await ctx.send(scores, 'high'); // Throws ZodError: Expected number, received string
```

## Back-Pressure

`ctx.send` (and `harness.send`) returns a `Promise<void>`. For value and topic channels — and queue channels below `capacity` — it resolves immediately. When a queue channel is full, the sender **parks** until a consumer dequeues an item:

- Parked senders resolve in FIFO order as `recv`/`tryRecv` free slots; the parked value lands at the queue tail.
- A parked send rejects with `channel_timeout` after 30 seconds (the value never enters the queue).
- Aborting the sending context rejects a parked send with `cancelled`.

External senders are different: `ChannelHandle.send` is synchronous and never blocks. At capacity it drops the **oldest** queued item (with a warning) so external callers like HTTP handlers can't be wedged by a full queue.

## Mode Comparison

| Behavior | Value | Queue | Topic |
|---|---|---|---|
| Buffered | Latest only | Up to capacity | Not buffered |
| Multiple readers | All get same value | Each gets unique item | All waiting readers get copy |
| `tryRecv` when empty | `null` | `null` | Always `null` |
| Backpressure | Overwrites | Internal sends park at capacity; external handle drops oldest | Drops if no listeners |
